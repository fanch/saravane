Comment construire un paquet ?

Introduction

Nous connaissons les raisons de contribuer, les connaissances requises, les commandes de base à notre disposition, le format minimum d'une recette et enfin quel sont les recettes existantes. Il est temps de passer à l'action et de construire son premier paquet dans les règles de l'art (de NuTyX) .

Quel scénario choisissez-vous ?

Qu'est-ce qu'un scénario dans le contexte d'une distribution linux ?

Ne demandez pas (encore) à google, je ne crois pas que ce concept existe déjà sur d'autres distributions

C'est un concept que j'ai imaginé en crééant le gestionnaire de paquets cards.

Au fil des années, je me suis rendu compte que le public cible pour l'utilisation d'une distribution comme NuTyX était très très large. Chaque utilisateur (plus ou moins averti) souhaite évoluer sur sa distribution à sa manière.

La plus part des utilisateurs se dirigeant vers NuTyX souhaitent compiler leurs paquets à leur guise. Certains ne choisiront peut-être pas toutes les sections proposées par défaut. D'autres utilisateurs souhaiteront mélanger les paquets standards proposés en binaire avec leur propres paquets. Toutes ces approchent sont différentes, je les appelle scénarios d'utilisation de la NuTyX.

Quels sont les scénarios possibles (cad ce que je peux imaginer aujourd'hui)

  1. Scénario A: Je ne veux rien compiler, les binaires disponibles me suffisent. Cela peut vous paraître absurde d'en parler ici, mais comme vous êtes très interessés par le sujet, sachez que ce scénario vous permet de garder votre distribution favorite la plus légère possible.
  2. Scénario B: Les binaires disponibles ne me suffisent pas, je voudrais ajouter un ou deux paquets.
  3. Scénario C: Je ne souhaite pas installer les binaires des sections console et desktop. Je les compile moi-même.
  4. Scénario D: Je souhaite utiliser les binaires disponibles des sections console et desktop. Je souhaite que mes ports personnels utilisent les dépendances provenant des binaires déjà disponibles. Seul mes ports personnels doivent être compilés.
  5. Scénario E: Je souhaite construire mes recettes dans un environnement isolé et garder ma distribution intacte et vierge de tout paquets de développement. Mes paquets produits dans mon environnement isolé pourront être installés dans ma NuTyX principale.

Ajuster les fichiers de configuration en fonction de votre scénario

Introduction

Voyons en détail chaque scénario mentionné plus haut et comment ajuster le mieux les fichiers de configuration pour chacun d'entre eux.

Scénario A

Introduction

Comme expliqué plus haut, vous ne souhaitez pas compilez dans votre NuTyX. Ce qui est tout à fait possible si vous n'avez pas besoin de programmes particuliers et que les binaires proposés vous suffisent. Ce scenario est aussi valable dans le cas où vous gérer plusieurs installation sur les quelles aucune compilation n'est necessaire.

Le fichier /etc/cards.conf

Les sections binaires sont activées

L'arch de la machine est vérfiée

La(les) langue(s) utilisée(s) sur la machine est ajustée(s)

Le dossier définissant le système de base, en cas de retour à un système de base est défini

Le contenu typique de votre fichier /etc/cards.conf:

dir /var/lib/pkg/saravane/desktop|http://downloads.nutyx.org/saravane/x86_64/desktop
dir /var/lib/pkg/saravane/console|http://downloads.nutyx.org/saravane/x86_64/console
dir /var/lib/pkg/saravane/base|http://downloads.nutyx.org/saravane/x86_64/base
arch x86_64
locale fr
locale de
base /var/lib/pkg/saravane/base

Les trois sections standards base, console et desktop sont activées. La machine est une 64 bits x86_64. Les langues qui doivent être installées sont le français fr et l'allemand de. Le dossier contenant les paquets de base est celui de la section standard base.

base /var/lib/pkg/saravane/base

Le fichier /etc/pkgmk.conf

Pour le scénario A, le fichier /etc/pkgmk.conf n'est pas présent puisque vous ne devez rien compiler. Les outils de développement ne sont pas présents.

Conclusion

C'est tout ce qu'il faut savoir pour la configuration de votre NuTyX. Vous souhaitez mettre à jour votre base de données et voir si il y a des maj disponibles

check

Vous souhaitez installer un paquet:

get firefox

Vous souhaitez supprimer un paquet:

del firefox

C'est tout ce qu'il faut savoir . J'espère que vous allez vous en sortir.

Scénario B

Introduction

Vous êtes satisfait de votre NuTyX, il y a juste un ou deux paquets qui vous manquent, vous décidez donc de les compilez vous-même.

Armez-vous de courage et de patience .

Un bon conseil, dans ce scénario, évitez à tout prix de compiler des recettes réclamant des dépendances non comprises dans la section base de votre NuTyX. Si tel est le cas, vous serez confrontez aux nombreuses erreurs liées aux dépendances devel non installées par défaut. Dans un tel cas, préférez le scenario C.

Installation des outils de développement

On peut maintenant commencer par installer les paquets de développement, cad un compilateur, les librairies C et C++ les outils autoconf, automake, etc etc.

cards sync
/var/lib/pkg/saravane/desktop/.PKGREPO
/var/lib/pkg/saravane/console/.PKGREPO
/var/lib/pkg/saravane/base/.PKGREPO
cards install cards.devel

Le fichier /etc/cards.conf

Un dossier contenant mes recettes personnelles est ajouté, ce dossier ne doit jamais être mis à jour par la commande cards sync, donc on ne doit pas spécifier d'adresse URL associées.

Les sections binaires sont activées, cela me permettra d'ajouter facilement les éventuelles dépendances manquantes...

L'arch de la machine est vérfiéei, nous sommes en 64 bits.

La(les) langue(s) utilisée(s) sur la machine est ajustée(s)

Le dossier définissant le système de base, en cas de retour à un système de base est défini

Le contenu typique de votre fichier /etc/cards.conf pour le scénario B:

dir /usr/ports/personnel
dir /var/lib/pkg/saravane/desktop|http://downloads.nutyx.org/saravane/x86_64/desktop
dir /var/lib/pkg/saravane/console|http://downloads.nutyx.org/saravane/x86_64/console
dir /var/lib/pkg/saravane/base|http://downloads.nutyx.org/saravane/x86_64/base

Le fichier /etc/pkgmk.conf

Dans ce scénario, il est bon de vérifier quelques variables:

Mon processeur est un 8 coeurs, donc autant en profiter:

export MAKEFLAGS="-j8"

Les sources doivent être stockées dans le dossier contenant la recette ou dans un dossier séparé. Personnellement, je préfère les mettre ensemble dans un dossier séparé. Et pourquoi pas un dossier qui sera nettoyé au prochain boot, je choisi le dossier /tmp. Je souhaite que les sources soient effacées une fois le paquet compilé et installé avec succès. Et enfin, je préfère que le dossier de travail soit également sous le dossier /tmp/work en lieu du dossier défini par défaut:

PKGMK_SOURCE_DIR="/tmp"
PKGMK_KEEP_SOURCES="no"
PKGMK_WORK_DIR="/tmp/work"

Je n'ai pas envie de spécifier à chaque fois la demande de téléchargement des sources si pas disponible. Une variable permet de le faire automatiquement:

PKGMK_DOWNLOAD="yes"

Dans ce scénario, je ne souhaites pas gérer les dépendances de façon automatique lors de la compilation et donc je n'ai pas besoin du fichier .PKGREPO... Je n'utilise que la commande pkgmk:

PKGMK_IGNORE_REPO="yes"
PKGMK_IGNORE_CATEGORY="yes"

Je ne souhaite pas que pkgmk split le binaire en paquet et sous paquets (devel, doc, man, etc). Je me contente du paquet de base puisque je le compile moi-même. Pour les locales, si elles existent, je me contenterai des locale françaises et allemandes fr et de: 

PKGMK_GROUPS=()
PKGMK_LOCALES=(fr de)

Je ne souhaite pas que pkgmk compresse mon paquet une fois construit et ne souhaite pas le conserver une fois installé:

PKGMK_COMPRESSION_PACKAGE="no"
PKGMK_CLEAN="yes"

Voici à quoi peut ressembler votre fichier /etc/pkgmk.conf pour le scénario B:

export CFLAGS="-O2 -pipe"
export CXXFLAGS="${CFLAGS}"
case ${PKGMK_ARCH} in
    "x86_64"|"")
       export MAKEFLAGS="-j8"
       ;;
    "i686")
       export CFLAGS="${CFLAGS} -m32"
       export CXXFLAGS="${CXXFLAGS} -m32"
       export LDFLAGS="${LDFLAGS} -m32"
       ;;
    *)
       echo "Unknown architecture selected! Exiting."
       exit 1
       ;;
esac
PKGMK_SOURCE_DIR="/tmp"
PKGMK_KEEP_SOURCES="no"
PKGMK_WORK_DIR="/tmp/work"
PKGMK_IGNORE_REPO="yes"
PKGMK_IGNORE_CATEGORY="yes"
PKGMK_GROUPS=()
PKGMK_LOCALES=(fr de)
PKGMK_COMPRESSION_PACKAGE="no"
PKGMK_CLEAN="yes"

Voilà pour ce qui est de la configuration du fichier /etc/pkgmk.conf

Conclusion

On verra plus tard, comment créer, compiler et installer un paquet sans dépendances ou dont les dépendances sont déjà installées. Pour ce scénario, la seule commande à retenir est: 

pkgmk
pkgadd

Cette commande sera toujours lancée dans le dossier de la recette.

Scénario C

Introduction

Vous avez des envies de compilation à grande échelle, vous êtes convaincu que c'est le meilleur scénario pour gérer une NuTyX. Vous n'avez aucune envie d'utiliser les binaires proposés.

La plus part des paquets que vous utilisés ne sont pas présent et ceux qui le sont, seront compiler par vous-même.

Vous souhaitez obtenir les maj des ports disponibles.

Et bien évidement, vous souhaitez que les dépendances soient compilées automatiquement.

Vous l'avez compris, ce scenario est sans doute le plus "attirant" et le plus proche des distributions "dite" sources telle que CRUX ou gentoo.

Le fichier /etc/cards.conf

Un dossier contenant mes recettes personnelles est ajouté, ce dossier ainsi que les sections ne doivent jamais être mis à jour par la commande cards sync, donc on ne spécifie aucune adresse URL.

Seule la section base doit rester active afin de pouvoir installer les outils de développement.

L'arch de la machine est vérfiée, nous sommes en 32 bits.

Le dossier contenant les logs de compilation est défini

Les dossiers définissant le système de base, en cas de retour à un système de base sont définis

Le contenu typique de votre fichier /etc/cards.conf pour le scénario C:

dir /usr/ports/personnel
dir /usr/ports/gnome
dir /usr/ports/desktop
dir /usr/ports/console
dir /usr/ports/base
dir /var/lib/pkg/saravane/base|http://downloads.nutyx.org/saravane/i686/base
arch i686
logdir /var/log/pkgbuild
locale fr
locale it
base /usr/ports/base
base /usr/ports/aconserver

Le fichier /etc/pkgmk.conf

Pour ce scénario, il est indispensable de vérifier toutes les variables necessaires pour le bon fonctionnement du duo pkgmk/cards

Mon processeur est un 4 coeurs mais je suis sur une NuTyX 32 bits.

export MAKEFLAGS="-j4"

Les sources sont stockées dans un dossier commun, je ne souhaite pas qu'elles soient effacées. Je souhaite que le dossier de travail reste dans le dossier du port, cela me permettra eventuellement de compiler plusieurs ports sans perdre le travail. C'est très utile pour vérifier.

PKGMK_SOURCE_DIR="/srv/www/sources"
PKGMK_KEEP_SOURCES="yes"
PKGMK_WORK_DIR="$PWD/work"

Je n'ai pas besoin de spécifier la demande de téléchargement des sources si pas disponible.

PKGMK_DOWNLOAD="no"

Dans ce scénario, je souhaite gérer les dépendances de façon automatique lors de la compilation et donc j'ai besoin du fichier .PKGREPO généré dans le dossier du port. Je souhaite pouvoir utiliser la commande cards install ou get pour pouvoir réinstaller mes binaires disponibles. La commande pkgmk doit donc générer le .PKGREPO de la section concernée (dossier parent). Je utilise le duo cards/pkgmk:

PKGMK_IGNORE_REPO="no"
PKGMK_IGNORE_CATEGORY="no"

Je ne souhaite pas que pkgmk split le binaire en paquet et sous paquets (devel, doc, man, etc). Je me contente du paquet de base puisque je le compile moi-même. Pour les locales, si elles existent, je me contenterai des locale françaises et italiennes fr et it: 

PKGMK_GROUPS=()
PKGMK_LOCALES=(fr it)

Je souhaite que pkgmk compresse mon paquet au format gz (parce que l'algo est beaucoup plus rapide qu'en xz ) une fois construit et qu'il soit conservé une fois installé pour usage ultérieur. Par exemple si je souhaite revenir à un système de base via la commande cards base:

PKGMK_COMPRESSION_PACKAGE="yes"
PKGMK_COMPRESSION_MODE="gz"
PKGMK_CLEAN="no"

Voici à quoi peut ressembler votre fichier /etc/pkgmk.conf pour le scénario C:

export CFLAGS="-O2 -pipe"
export CXXFLAGS="${CFLAGS}"
case ${PKGMK_ARCH} in
    "x86_64"|"")
       export MAKEFLAGS="-j4"
       ;;
    "i686")
       export CFLAGS="${CFLAGS} -m32"
       export CXXFLAGS="${CXXFLAGS} -m32"
       export LDFLAGS="${LDFLAGS} -m32"
       export MAKEFLAGS="-j4"
       ;;
    *)
       echo "Unknown architecture selected! Exiting."
       exit 1
       ;;
esac
PKGMK_SOURCE_DIR="/srv/www/sources"
PKGMK_KEEP_SOURCES="yes"
PKGMK_WORK_DIR="$PWD/work"
PKGMK_IGNORE_REPO="no"
PKGMK_IGNORE_CATEGORY="no"
PKGMK_GROUPS=()
PKGMK_LOCALES=(fr it)
PKGMK_COMPRESSION_PACKAGE="yes"
PKGMK_COMPRESSION_MODE="gz"
PKGMK_CLEAN="no"

Scénario D

Introduction

Vous n'avez pas envie de vouloir tout compiler, vous êtes satisfait avec les binaires disponibles. Vos nouvelles recettes utilisent les dépendances provenant des binaires existants. Ce scénario est probablement le plus long à mettre en oeuvre mais une fois tout en place, l'ajout de paquets personnel se fait très rapidement.

Le fichier /etc/cards.conf

Un dossier contenant mes recettes personnelles est ajouté, ce dossier ainsi que les sections ne doivent jamais être mis à jour par la commande cards sync. On ne spécifie aucune URL.

Seule la section base doit rester active le temps de récupérer la commande rsync si pas encore installée.

L'arch de la machine est vérfiée, nous sommes en 64 bits.

Le dossier contenant les logs de compilation est défini

Je n'ai pas besoin de spécifier les locales, elles seront toujours toutes installées

Les dossiers définissant le système de base, en cas de retour à un système de base sont définis

Le contenu typique de votre fichier /etc/cards.conf pour le scénario D: 

dir /srv/www/saravane/personnel
dir /srv/www/saravane/x86_64/desktop
dir /srv/www/saravane/x86_64/console
dir /srv/www/saravane/x86_64/base

arch x86_64
logdir /var/log/pkgbuild
base /srv/www/saravane/x86_64/base
base /srv/www/saravane/aconserver

Installation de rsync et suppression du dossier concerné

Pour ce scénario, nous allons synchroniser tous les binaires disponible sur le miroir de NuTyX et allons restés "callés" sur la version de ceux-ci.

Commençons par installer la commande rsync si ce n'est pas encore le cas:

get rsync

On peut maintenant supprimer la ligne pointant sur l'url du miroir de NuTyX pour déactiver définitivement la commande cards sync.

Le nouveau fichier /etc/cards.conf:

Synchronisation des binaires via rsync

Cette opération consiste à rappatrier les quelques 11000 paquets disponible en binaires. Elle peut durer entre quelques minutes et quelques heures en fonction de votre connection.

Heureusement, cette opération n'est longue que la première fois .

Notez bien le dossier de destination, j'ai choisi de "déposer" les binaires à la racine du serveur http .

Il faut commencer par créer les dossiers

sudo mkdir -p /srv/www/saravane/{x86_64,personnel,aconserver}

On peut maintenant synchroniser l'ensemble des binaires:

sudo rsync -avz --delete-after rsync://downloads.nutyx.org/nutyx/saravane/x86_64/ \
/srv/www/saravane/x86_64/

Allez peut-être boire un café, manger une pizza ou un bon paquet de frites , ça peut-être long...

Pour ce scénario, il est indispensable de vérifier toutes les variables necessaires pour le bon fonctionnement du duo pkgmk/cards.

PKGMK_SOURCE_DIR="/srv/www/sources"
PKGMK_KEEP_SOURCES="yes"
PKGMK_WORK_DIR="/tmp/work"

Je n'ai pas besoin de spécifier la demande de téléchargement des sources si pas disponible.

Dans ce scénario, je souhaite que les dépendances soient installées mais non compilées. J'ai besoin du fichier .PKGREPO généré dans le dossier du port. Je souhaite pouvoir utiliser la commande cards install ou get pour pouvoir réinstaller mes binaires disponibles. La commande pkgmk doit donc générer le .PKGREPO de ma section personnelle.

PKGMK_IGNORE_REPO="no"
PKGMK_IGNORE_CATEGORY="no"

Puisque les dépendances proviennent des binaires exisantes et que ces binaires sont splités. Il est préférable de concerver cette fonctionnalitée pour les paquets personnels. Les locales produites seront les 11 locales produites par défaut.

PKGMK_GROUPS=(devel man doc service)
PKGMK_LOCALES=(fr de it es nl pt da nn sv fi)

Les binaires produits seront dans le même format de compression que les binaires officielles, même si ce n'est pas une obligation, on veut garder une certaine cohérence entre les binaires. Je ne souhaite pas supprimer mes binaires produits une fois installés.

PKGMK_COMPRESSION_PACKAGE="yes"
PKGMK_COMPRESSION_MODE="gz"
PKGMK_CLEAN="no"

Voici à quoi peut ressembler votre fichier /etc/pkgmk.conf pour le scénario D: 

export CFLAGS="-O2 -pipe"

case ${PKGMK_ARCH} in
    "x86_64"|"")
       export MAKEFLAGS="-j2"
       ;;
    "i686")
       export CFLAGS="${CFLAGS} -m32"
       export CXXFLAGS="${CXXFLAGS} -m32"

PKGMK_GROUPS=(devel man doc service)

PKGMK_COMPRESSION_MODE="xz"

Conclusion

Pour ce scénario, les commandes à retenir sont:

cards level
cards base -r
cards create
rsync -avz --delete-after rsync://downloads.nutyx.org/nutyx/saravane/x86_64/ \
/srv/www/saravane/x86_64/

Scénario E

Introduction

Ce scénario est un peu l'aboutissement de mon travail. Ici on propose un mélange de différents scénarios. On installe une NuTyX dans un dossier dédié dans laquelle on produit nos paquets personnels. Cela signifie que l'on doit parler de DEUX NuTyX

Ce scénario est donc la combinaison de DEUX scénario, dont un sera ... le scénario A ( maintenant vous savez pourquoi j'ai tenu à en parler ). Le deuxième scénario qui est celui de votre environnement isolé celui que vous préféré entre les 3 autres (B, C et D). A partir de maintenant on ne parle plus d'environnement isolée mais de la chroot.

Quelques avantages par rapports aux précédents:

  1. Comme vous n'installerez jamais de paquets de developpement sur votre NuTyX principale, celle ci gardera une taille raisonnable.
  2. Bien que vous devez prévoir quelques Giga pour votre chroot, sachez que si vous devez la supprimer, un simple: 
    rm -r /<chemin-de-votre-chroot>
    et vous retrouvez votre NuTyX principale.
  3. Tous vos paquets seront donc compilé dans la chroot, ce qui épargne toute fausse manoeuvre sur votre NuTyX principale.
  4. Votre chroot peut être un système de base, c'est même vivement recommandé alors que sur votre principale, vous utilisez très probablement une interface graphique.

Ma machine est une 64 bits à 4 coeurs

Les fichiers /etc/cards.conf

dir /var/lib/pkg/saravane/base|http://downloads.nutyx.org/saravane/x86_64/base
arch x86_64
locale fr
